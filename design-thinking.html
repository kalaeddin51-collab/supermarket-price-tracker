Phase 1 — Empathise

Understanding the User

Research, observation and immersion into the lived experience of North Sydney grocery shoppers.

The North Sydney Grocery Landscape

North Sydney is a dense, urban pocket with a mix of commuters, young professionals, and families. The area is served by multiple competing supermarket chains within walking distance of each other — Woolworths (Miller St), Coles (North Sydney), Harris Farm Markets (Crows Nest), and multiple IGA stores (Milsons Point, Cammeray, Greenwich). Residents are price-aware but time-poor, and the high cost of living in inner Sydney makes grocery costs a genuine household concern.

Key Empathy Insights

⏱️
Time-poor, not price-indifferent
Users care about price but can't invest time in manual comparison. Automation is the key enabler.
📍
Hyperlocal trust
Users only trust prices from stores they can physically reach. National comparisons are irrelevant.
🔔
Pull vs Push
Younger users want push alerts; older users prefer email digests. Both must be supported.

Phase 2 — Define

Framing the Problem

Synthesising research into actionable problem statements and design challenges.

Point of View Statement

North Sydney residents who shop across multiple local supermarkets need a reliable, automated way to monitor grocery prices across Woolworths, Coles, Harris Farm, and IGA simultaneously, because prices fluctuate frequently, stores don't make cross-store comparison easy, and manually checking prices is too time-consuming for busy households.

How Might We… Questions

help users know instantly when a tracked product drops below their target price, without them needing to check any website?
make cross-store price comparison feel as effortless as checking the weather?
ensure users only receive notifications when they're actually meaningful — not just noise?
support both tech-savvy and low-tech users with the same system, without compromise?
help users discover that a product they regularly buy is cheaper at a different nearby store?

Design Principles (Decided)

01 — No noise

Alerts only when meaningful

The system never sends emails or push notifications unless a genuine condition is met. No daily email spam.

02 — Zero install

Works in any browser

No native app, no Electron, no extension. A local web server accessible via browser is the only requirement.

03 — Local-first

Data stays on your machine

SQLite database, no cloud account required. User data never leaves the household.

04 — Hyperlocal

North Sydney area only

Five specific stores. Not a national aggregator — trust comes from local, verifiable accuracy.

05 — Progressive disclosure

Simple default, power when needed

The search page is a single input. Advanced filters, brand filtering, and price ranges reveal on demand.

06 — Graceful failure

One store down ≠ broken app

If a scraper fails, results from other stores still appear. No full-page errors.

Phase 5 — Test

Learning from Reality

Bugs found, fixes applied, and UX lessons learned during the build and test phase.

Issues Found & Resolved

✅

Search returning 0 results (Alpine hidden input race condition)

HTMX fired before Alpine.js had initialised the hidden store input, sending store="" which matched nothing. Fixed using x-effect for reactive sync and ID-based hx-include, plus server-side fallback to "all".

✅

Store enum AttributeError in Jinja2 templates

Template helpers called .title() on SQLAlchemy enum objects which don't have that method. Fixed with a _store_key() helper that extracts the string value safely.

✅

Test email showing success but displaying error

Email was delivered but Windows' charmap codec couldn't encode the 🛒 emoji in the subject line, causing the function to return False. Fixed by setting PYTHONIOENCODING=utf-8 and using plain-text subjects.

✅

Brand filter not working — quoting conflict in Alpine

{{ b | tojson }} produced double-quoted strings inside double-quoted HTML attributes, breaking the JavaScript. Fixed using data-brand="{{ b }}" attributes and reading $el.dataset.brand in the click handler.

✅

Brand filter only searching visible page, not all results

Client-side x-show filtering only operated on the 25 results on the current page. Redesigned as a server-side filter — brand selection triggers a new HTMX request with the brand parameter included.

✅

Brand filter clicks not registering

The #search-results div was outside the Alpine component that managed brand state. HTMX-swapped content had no Alpine scope, so @click directives were never processed. Fixed by moving results div inside the Alpine component boundary.

⚠️

IGA Crows Nest doesn't exist on igashop.com.au

The Crows Nest IGA is not listed in the Metcash platform. Mapped to IGA Greenwich (store_id=19133) as the nearest alternative (~1.2km). Labelled clearly in the UI as "IGA Greenwich (near Crows Nest)".

⚠️

Harris Farm Shopify API limited to 10 results per call

Shopify's predictive search endpoint hard-limits results to 10 items regardless of the limit parameter. Acceptable for watchlist-style use cases — documented in the scraper code.
